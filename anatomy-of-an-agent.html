The Anatomy of an AI Agent

Three acts, 300 lines of Python, and the architecture behind every coding agent.

There's a mystique around AI agents. They seem to think, plan, and act autonomously. They read your code, propose changes, run your tests. They feel like a leap beyond chatbots.

They're not. An AI agent is a while loop, a list of messages, and a set of tools. That's the entire architecture. Let's build one from scratch — in three acts.

Act 1

The Loop

The simplest possible agent is an API client, a messages list, and a loop. Here it is — the whole thing:

agent.py 
from openai import OpenAI

client = OpenAI()
messages = []

while True:
    user_input = input("You: ")
    messages.append({"role": "user", "content": user_input})

    response = client.chat.completions.create(
        model="gpt-4o", messages=messages
    )
    reply = response.choices[0].message
    messages.append(reply)
    print(f"Agent: {reply.content}")

This is already a conversation. Each message — yours and the model's — gets appended to the messages list. The model sees the entire history on every call. It remembers context, follows threads, builds on previous answers.

But it can only talk. It can't read your files. It can't run your code. It can't do anything. It's a chatbot with a good memory — nothing more.

Why it matters

This 12-line loop is the skeleton of every AI agent — from ChatGPT to Claude Code to Devin. Everything else is just adding tools.

A note on SDKs

This article uses the OpenAI SDK because it works with multiple providers — OpenAI, Gemini, Ollama, and Anthropic — just by changing the base URL. If you prefer the Anthropic SDK directly, the pattern is nearly identical. See our workshop materials for both versions.

Act 2

Teaching It to See

To give the model eyes, we give it tools. A tool is just a function with a JSON schema describing its name, purpose, and parameters. The model is trained to recognize these schemas and emit structured calls when it wants to use one.

Let's add two tools: read_file and list_files.

tool definition 
TOOLS = [{
    "type": "function",
    "function": {
        "name": "read_file",
        "description": "Read the contents of a file",
        "parameters": {
            "type": "object",
            "properties": {
                "path": {"type": "string"}
            },
            "required": ["path"]
        }
    }
}]

But defining tools isn't enough. The model might ask to use a tool, but someone has to actually run it. That's what the inner tool loop does — it checks whether the model wants to call a tool, executes it, and feeds the result back:

the tool loop 
while True:
    response = client.chat.completions.create(
        model="gpt-4o", messages=messages, tools=TOOLS
    )
    message = response.choices[0].message
    messages.append(message)

    if message.tool_calls:
        for tool_call in message.tool_calls:
            result = execute_tool(tool_call)
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": result,
            })
    else:
        print(message.content)
        break

Now the model can explore. Ask it "what files are in this directory?" and it will call list_files, read the results, and respond intelligently. Ask it to read your configuration file and it will. The model decides when to use tools — you just execute what it asks for.

This is the critical insight: the model doesn't see your filesystem. It receives tool results as text, reasons about them, and decides what to do next. It's reading, not seeing.

Why it matters

The model doesn't "see" your filesystem. It reads tool results as text, reasons about them, and decides what to do next. This is the same pattern whether the tool reads a file, queries a database, or calls an API.

Act 3

Teaching It to Act

Seeing is useful. But the leap from assistant to agent happens when the model can change things. Two more tools complete the picture: edit_file replaces a specific string in a file (forcing the model to read first, so it knows the exact content). run_bash executes shell commands — with safety checks and a timeout.

With these four tools, the agent can read code, edit it, and run tests to verify its changes. Watch what happens when you give it a real task:

agent session 
You: Add error handling to the divide function in math.py

  [tool] read_file({"path": "math.py"})
  [tool] edit_file({"path": "math.py",
         "old_string": "return a / b",
         "new_string": "if b == 0:\n        return 'Error: division by zero'\n    return a / b"})
  [tool] run_bash({"command": "python -m pytest test_math.py"})

Agent: I've added a zero-division check to the divide
       function. All tests pass.

The model read the file, figured out the exact string to replace, made the edit, ran the tests, and reported back. Four tool calls, fully autonomous, no human in the loop.

That's the complete architecture. A while loop on the outside for the conversation. A while loop on the inside for tool calls. A handful of tool definitions and implementations. In about 300 lines of Python, you have a working coding agent.

Why it matters

In 300 lines of Python, you have a coding agent. Not a toy — the same architecture that powers production tools. The difference between this and Claude Code is polish, not paradigm.

An AI agent isn't magic. It's a while loop, a list of messages, and a set of tools. The magic is in the model — trained on the sum of human code to make reasonable decisions about what to read, what to change, and when to stop. Understanding this architecture is the first step to building AI tools that actually work.
